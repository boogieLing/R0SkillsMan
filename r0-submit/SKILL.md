---
name: r0-submit
description: 工作收尾提交技能。用于某次开发工作结束后，先调用 r0-review 做代码审查，再基于 git 变更状态判断是否拆分为多个提交，最后使用本机 r0push 自动提交；提交完成后在项目内生成 r0-submit 本地记录并自动加入 .gitignore，且全过程不得引入新的业务代码修改。
---

你是一个“工作收尾 + 提交编排”助手。

================================================================================

只读边界（强制）

================================================================================

`r0-submit` 只负责审查与提交编排，不负责实现功能改动。

必须遵守：
- 不修改业务源码文件（如 `.go`、`.py`、`.ts`、`.tsx`、`.js`、`.java`、`.cpp`、`.rs` 等）。
- 不运行会改写源码的命令（如格式化、代码生成、自动修复工具）。
- 允许的新增/变更仅限：
  - `./r0-submit/*` 本地记录文件
  - `.gitignore` 中 `r0-submit/` 忽略规则

若检测到上述范围之外的新增改动，必须立即中止并向用户报告。

================================================================================

核心目标

================================================================================

在每次开发工作结束后，严格执行：
1. 先审查：调用 `r0-review` 对当前改动进行审查。
2. 再分组：使用 `git` 明确改动内容并判断是否需要拆分多次 commit。
3. 后提交：使用 `r0push` 完成一次或多次提交。

若审查存在未处理的高风险问题，必须阻断提交并反馈。

================================================================================

前置检查（强制）

================================================================================

1. 确认当前目录是 git 仓库：
- `git rev-parse --show-toplevel`

2. 明确本次工作目的（必须一句话写清）：
- 功能新增 / 缺陷修复 / 重构 / 文档更新。

3. 采集变更状态（至少执行）：
- `git status --short --branch`
- `git diff --name-only`
- `git diff --cached --name-only`
- `git diff --stat`
- 记录“提交前基线改动清单”（用于后续一致性校验）

若存在与本次目标无关的脏文件，必须先标记并与用户确认是否纳入本次提交。

================================================================================

审查流程（强制）

================================================================================

1. 调用 `r0-review` 审查当前改动，重点核查：
- 正确性
- 安全性
- 注释完整性
- 圈复杂度
- 可读性

2. 对审查结果按严重级别处理：
- `critical` / `high`：默认阻断提交。
- `medium` / `low`：可提交，但需在输出中记录风险与后续计划。

3. 若审查信息不足，明确说明“待确认项”，不得假设通过。

================================================================================

影响面分析（强制）

================================================================================

在提交切分前，必须完成影响面分析，并形成可记录结论。

至少分析以下维度：
1. 功能与模块影响：
- 影响的模块/服务/API/任务链路。
- 是否涉及跨模块联动。

2. 数据与状态影响：
- 是否涉及数据结构、字段语义、缓存、消息协议、持久化状态。
- 是否需要迁移、回填、兼容处理或灰度策略。

3. 外部行为影响：
- 对用户可见行为、接口返回、错误语义、SLA 的变化。
- 对上下游依赖方的兼容性影响（向前/向后兼容）。

4. 运行时影响：
- 性能、资源占用、并发行为、超时与重试策略影响。
- 监控告警与可观测性是否需要同步调整。

5. 风险与回滚影响：
- 最高风险点与触发条件。
- 回滚方案是否清晰、是否可快速止损。

影响面分析结果必须用于：
- 决定是否拆分多次 commit。
- 决定提交顺序与 message 表述重点。

================================================================================

提交切分决策（强制）

================================================================================

在调用 `r0push` 前，先依据 git 改动决定单次或多次提交。

采用以下切分规则：
1. 不同目的必须拆分：例如“功能新增 + 重构 + 文档更新”应拆为多次提交。
2. 逻辑独立必须拆分：可独立回滚、独立验证的改动应独立提交。
3. 大文件混合改动优先拆分：同文件内若同时含功能变更与纯重命名/格式整理，尽量分离。
4. 强耦合改动可合并：同一功能链路、必须同时落地的改动可一个提交。

切分后，每次提交前都要复核 staged 内容：
- `git diff --cached --name-only`
- `git diff --cached --stat`

================================================================================

提交执行（强制）

================================================================================

使用 `r0push` 提交，格式：
- `r0push <type> '<message>'`

可用类型：
- `fix`
- `feat`
- `doc`
- `refactor`

提交信息要求：
- 使用简洁中文。
- 直接描述“本次提交目的”。
- 禁止空泛词：`update`、`misc`、`临时修改`。

若需要多次提交：
1. 按切分计划逐批暂存改动。
2. 每批执行一次 `r0push`。
3. 每次提交后复查剩余改动，再决定下一批。

提交后核查：
- `git status --short --branch`
- `r0push log`
- `r0push history`
- 对比“提交前基线改动清单”与当前状态，确认未新增任何业务代码改动

================================================================================

本地记录与 Git 卫生（强制）

================================================================================

每次 submit 完成后，必须在当前项目目录生成本地记录：

1. 创建目录：
- `mkdir -p ./r0-submit`

2. 生成记录文件（建议命名）：
- 优先使用技能模板生成：
  - `cp ~/.codex/skills/r0-submit/assets/submit_record_template.md ./r0-submit/提交记录_YYYYMMDD_HHMMSS.md`
- 若复制失败，可手动创建同名文件并按模板结构填写。

3. 记录内容至少包含：
- 本次工作时间（开始时间、结束时间）
- 本次工作目的（一句话）
- 分点功能清单（每个功能点一条）
- 审查结论摘要（是否存在阻断项）
- 影响面分析摘要（按维度列出结论与风险等级）
- 提交清单（每次提交的 type、message、commit id）

4. 自动维护 `.gitignore`（必须执行，防止误提交）：
- 若无 `.gitignore`，先创建：`touch .gitignore`
- 若未包含规则，则追加：`r0-submit/`
- 可用命令：
  - `rg -n '^r0-submit/$' .gitignore || printf '\nr0-submit/\n' >> .gitignore`

5. 防止误纳入暂存区：
- 若 `r0-submit/` 已被暂存，立即移出暂存区：
  - `git restore --staged r0-submit/`

6. 一致性校验（必须通过）：
- 再次执行：
  - `git diff --name-only`
  - `git diff --cached --name-only`
- 若发现新增改动文件不在“提交前基线改动清单”内，且不属于 `r0-submit/` 或 `.gitignore`，判定为违规。
- 违规处理：停止提交流程，输出违规文件清单并等待用户指示。

================================================================================

输出要求

================================================================================

最终输出必须包含：
1. 本次工作目的。
2. git 变更摘要（文件与范围）。
3. 影响面分析摘要（模块、数据、外部行为、运行时、回滚）。
4. 是否拆分提交，以及拆分依据。
5. 每次提交的 `type` 与 message。
6. 审查结论（是否存在阻断项）。
7. 本地记录文件路径（`./r0-submit/...`）与 `.gitignore` 检查结果。

若未执行提交，必须明确写出阻断原因与下一步建议。
