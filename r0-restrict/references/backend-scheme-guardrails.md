# Backend Scheme Guardrails

## 使用方式
- 当任务已经确认是后端技术方案、架构评审、后端实现门禁或后端 review 时再读取本文件。
- 本文件是 `r0-restrict` 的详细规则库；不要把整份内容原样塞回宿主输出，应提炼成约束、门禁与验证项。

## 1. 三维总框架

### 数据流转
- 先画出最短业务主链：请求入口 -> 参数校验 -> 鉴权 -> 读写路径 -> 异步路径 -> 回调/补偿。
- 对每一跳回答：
  - 数据从哪里来、写到哪里去
  - 是同步还是异步
  - 是否会回源
  - 失败后谁补偿
  - 是否会把用户阻塞在主路径上

### IO 操作
- 统计主路径与异步路径的 IO 次数与类型：
  - MySQL
  - Redis
  - RPC / HTTP
  - MQ
  - 定时任务 / 扫描任务
- 目标是用最少 IO 完成业务，并让高频路径可批量、可降级、可限流。

### 组件依赖
- 列出：
  - 上游入口
  - 当前服务
  - 强依赖
  - 弱依赖
  - 降级路径
- 识别依赖故障会不会放大为雪崩或用户侧明显抖动。

## 2. 数据库与 IO 操作规范

### 2.1 循环内 IO
- 严禁在 `for` 循环中发起 MySQL、Redis、RPC 等 IO。
- 一律优先改为批量操作：
  - Redis `MGET` / `HMGET`
  - SQL `IN` / 批量写
  - 批量 RPC 接口
- 若暂时无法批量化，必须明确这是阻塞风险，并说明后续收敛路径。

### 2.2 扫库 / 扫缓存
- 慎用大范围扫库、扫缓存。
- 避免无索引查询、全表扫描、全量 Redis 扫描。
- 必要时必须同时具备：
  - 低峰期执行
  - cursor 或 limit
  - 单批上限
  - 可中断与可续跑能力

### 2.3 读写策略
- 明确读写比，决定索引与存储结构。
- 高频同步写要优先评估：
  - 异步写
  - 批量写
  - 先写 MQ / Redis 再批量落库
- 不要在流量高峰下把大量同步写直接压到 MySQL。

## 3. Redis 设计与防坑

### 3.1 TTL
- Redis Key 默认必须有合理 TTL。
- 只有极少数配置字典类数据才允许长期不过期。
- TTL 过短也不行，会造成频繁回源。

### 3.2 热点 Key 与击穿
- 高频热点 Key 禁止通过 `Delete` 后懒加载方式失效。
- 优先使用：
  - 主动 `Update`
  - 逻辑过期
  - 分布式互斥锁
  - 单线程回源更新

### 3.3 BigKey
- 选数据结构前必须预估数据量。
- 避免 BigKey 拉高 Redis CPU、网络和迁移成本。
- 需要拆分时优先按业务维度或时间维度拆 Key。

### 3.4 Pipeline / Lua
- 多条无依赖命令优先考虑 Pipeline。
- 需要原子性时优先考虑 Lua。
- 不要把多次 RTT 明明可以合并的 Redis 操作分散发送。

## 4. 分布式与高可用健壮性

### 4.1 分布式锁
- 必须设置可靠的超时释放时间。
- 必须回答两个问题：
  - 业务执行超时导致锁提前释放怎么办
  - Pod 重启 / 宕机后遗留业务和锁怎么兜底
- 常见兜底：
  - WatchDog 自动续期
  - 补偿任务
  - 幂等重放

### 4.2 幂等性
- MQ 消费、定时任务、外部回调等核心变更逻辑必须天然支持重复执行。
- 不接受“理论上只会执行一次”这种假设。

### 4.3 执行时长与重试
- 定时任务和 MQ 消费必须评估：
  - 单次执行时长
  - 并发度
  - 重试间隔
  - 最大重试次数
  - 死信 / 告警 / 人工介入策略

## 5. To-C 场景补充门禁

### 5.1 高并发与流量防护
- 必须评估限流、降级、熔断三板斧。
- 限流要回答：
  - QPS 上限是多少
  - 在网关还是应用层做
  - 超限时如何友好返回
- 降级要回答：
  - 哪些链路是非核心链路
  - 高峰时可以先关什么
- 熔断要回答：
  - 哪些下游是高风险依赖
  - 响应慢或报错时如何快速止损

### 5.2 多级缓存与 RT
- 面向用户的核心接口要关注 TP99，通常目标在 200ms 甚至 100ms 以内。
- 极高并发且变更不频繁的数据，优先考虑：
  - 本地缓存
  - Redis 分布式缓存
  - DB
- 能异步返回的，不要让用户同步等待。

### 5.3 安全与防刷
- 涉及资产、注册、抽奖等链路，必须评估风控或人机验证。
- 服务端必须自行从 Token 还原 `userId`，不要信任客户端透传。
- 关键接口要评估防重放：
  - nonce
  - timestamp
  - 签名校验
- 日志与前端展示必须脱敏手机号、身份证、银行卡等敏感信息。

### 5.4 版本兼容
- App / 小程序用户长期不升级是常态。
- 已发布接口默认向前兼容：
  - 不随意删字段
  - 不随意改字段语义
  - 新增必填要有默认值
  - 需要时做 `/v1` / `/v2`

### 5.5 可观测性
- TraceId 必须贯穿主链路。
- 关键指标至少包括：
  - 错误率
  - RT
  - 核心业务指标
  - 任务堆积
- 要保证客诉场景能根据用户标识快速反查完整链路。
